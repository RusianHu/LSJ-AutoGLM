# P1. 项目定义与边界

## 1.1 背景

当前项目已经具备可用的 TUI/CLI 能力：

- 交互式启动器主要位于 [launcher.py](../launcher.py)
- 主执行入口位于 [main.py](../main.py)
- Agent 主循环位于 [phone_agent/agent.py](../phone_agent/agent.py)
- 动作执行位于 [phone_agent/actions/handler.py](../phone_agent/actions/handler.py)
- Android 截图位于 [phone_agent/adb/screenshot.py](../phone_agent/adb/screenshot.py)
- Android 设备控制位于 [phone_agent/adb/device.py](../phone_agent/adb/device.py)
- ADB 连接管理位于 [phone_agent/adb/connection.py](../phone_agent/adb/connection.py)

现状可以稳定完成：设备检查、API 配置、任务执行、日志输出、人工输入式确认/接管。

现状不足也很明确：

- 主要交互依赖终端
- 运行过程缺少桌面化可视化承载
- 当前“看屏”和“控制”分离，无法直接形成桌面工作台体验
- 确认/接管机制仍是控制台输入，不适合 GUI 工作流

## 1.2 已锁定的产品方向

基于访谈，首版 GUI 方向锁定如下：

- [x] 产品角色：全功能客户端
- [x] 演进方式：薄壳包装现有 Python 脚本与模块
- [x] 桌面技术：PySide6 / Qt
- [x] 目标用户：以你本人高频自用为主
- [x] 首版运行环境：Windows 自用版
- [x] 交付方式：源码运行优先
- [x] 导航模型：左侧多页导航
- [x] 配置策略：延续 `.env` / 环境变量为主
- [x] 任务执行方式：独立子进程运行
- [x] 实时看屏方向：内嵌 scrcpy 类方案

## 1.3 首版目标

### 产品目标

- [ ] 将现有 TUI 主要价值迁移到桌面 GUI 中
- [ ] 建立“设备与镜像 + 日志与事件”为中心的主工作台
- [ ] 支持从 GUI 发起任务、停止任务、查看实时输出和最终结果
- [ ] 支持人工接管：暂停后由你操作，再继续执行
- [ ] 支持保存任务历史与完整原始日志
- [ ] 支持记录关键事件、性能指标、环境快照和错误上下文

### 工程目标

- [ ] 尽量复用现有 [launcher.py](../launcher.py) 与 [main.py](../main.py) 的配置和执行逻辑
- [ ] 尽量不破坏当前 CLI/TUI 可用性
- [ ] 通过新增 GUI 层逐步替换终端交互，而不是一开始就重构全部底层
- [ ] 为后续更深层服务化或调试化预留边界

## 1.4 首版非目标

以下内容明确不进入首版范围：

- [ ] 多任务队列 / 编排面板
- [ ] 单步调试器
- [ ] 模型上下文透视面板
- [ ] iOS 完整 GUI 支持
- [ ] HarmonyOS 完整 GUI 支持
- [ ] 多设备并行镜像与并行任务
- [ ] 面向陌生用户的通用发行级易用性优化

## 1.5 设计原则

- [ ] 优先做“能每天用”的桌面工作台，而不是抽象完美的框架
- [ ] 优先复用现有执行链，避免首版过早重构 Agent 内核
- [ ] UI 决策以你的实际使用频率和信息密度为中心
- [ ] 首版先跑通 Android + ADB + Windows 的最强主路径
- [ ] 功能优先级高于包装与跨平台泛化

# P2. 用户场景与核心工作流

## 2.1 主要用户画像

### 核心用户

- [x] 单一主用户：你本人
- [x] 熟悉项目结构、ADB、模型配置与本地开发环境
- [x] 需要高效率切换设备、模型、任务与日志视图
- [x] 更在意“工作台效率”和“运行透明度”，而不是纯小白向引导

### 由此带来的直接设计含义

- [ ] 可以保留一定专业术语与配置透明度
- [ ] 可以接受信息密度较高的界面
- [ ] 可以优先服务单设备主工作流
- [ ] 可以优先暴露日志与状态，不必过度做新手向封装

## 2.2 核心场景

### 场景 A：快速开始一次任务

1. 打开 GUI
2. 看到当前设备连接状态与手机镜像
3. 检查当前模型/API/profile
4. 输入任务
5. 点击开始
6. 在日志与事件区观察运行情况
7. 任务完成后查看结果与历史记录

### 场景 B：任务中遇到验证码/登录/异常页面

1. GUI 检测到接管或失败征兆
2. 任务进入暂停态
3. 手机镜像继续可见
4. 你手动操作手机完成必要步骤
5. 点击“继续执行”
6. Agent 继续后续流程

### 场景 C：排查失败

1. 在任务历史中打开失败记录
2. 查看原始日志
3. 查看关键事件标记
4. 查看环境快照
5. 查看最后错误上下文
6. 决定是否重跑或调整配置

### 场景 D：切换不同模型/预设

1. 在 GUI 设置或 profile 区切换 `.env` 驱动的配置
2. 对 base URL、模型名、API Key 来源进行可视化校验
3. 重新发起任务

- [ ] 镜像区域承载

### 第 3 层：现有执行链

## 5.3 子进程执行模型

首版建议采用“GUI 主进程 + 任务子进程”的稳定结构。

### 设计要求

- [ ] 每次任务运行创建独立子进程
- [ ] GUI 捕获 stdout / stderr
- [ ] GUI 维护任务状态：idle / running / paused / stopping / failed / completed
- [ ] GUI 记录退出码、结束时间、错误摘要

### 这样做的原因

- [ ] 可以隔离模型请求或设备操作带来的崩溃影响
- [ ] 便于沿用现有 `print()` 输出
- [ ] 便于后续逐步把 stdout 升级为结构化事件
- [ ] GUI 自身保持稳定，不因任务异常退出

## 5.4 与现有代码的接口策略

### 首版建议

- [ ] 配置读取：优先复用 `.env` 约定
- [ ] 任务运行：优先通过命令行参数启动 [main.py](../main.py)
- [ ] 设备检查：GUI 内可直接复用已有 ADB 连接与检查逻辑

- [ ] 为 GUI 与 CLI 共享能力做准备

- [ ] 在 GUI 中嵌入或承载镜像窗口
- [ ] 响应窗口重建、设备断连、镜像退出

### 正常运行时

- [ ] 镜像主要承担可视化与观察

# P10. 风险、权衡与后续版本

## 10.1 首版关键风险

### 风险 A：scrcpy 类内嵌复杂度高于预期

影响：

- [ ] 可能拖慢首版落地

应对：

- [ ] 保留降级方案：外部镜像窗口或 ADB 截图预览模式
- [ ] 先保证工作台与任务链路成立，再追求镜像内嵌完美体验

### 风险 B：子进程暂停 / 恢复不够优雅

影响：

- [ ] 接管体验可能不顺滑

应对：

- [ ] 首版允许“任务暂停 + 你操作 + 继续/重跑”这样较保守的恢复策略

### 风险 C：现有输出以 print 为主，结构化程度不足

影响：

- [ ] 事件抽取成本上升

应对：

- [ ] 首版先做日志捕获与高层事件推断
- [ ] V2 再推动结构化事件输出
